\subsection{Plugin System}
\label{sec:sys-plugins}

% Core provides only two interfaces: graph read (plugin gets full obj/mor data including sort, content fields) and result write-back (plugin writes computation results to node/edge metadata fields or maps to visual properties)
%
% Two plugin categories, sharing same infrastructure (.astrolabe/plugins/, Python module dynamic FastAPI router registration, frontend skill dynamic loading):
%
% Data import plugins (direction: external → graph):
% - ilean parser plugin: Lean compilation artifacts → obj/mor (first plugin, validates architecture)
% - BibTeX import: citation networks → obj/mor
% - LaTeX structure extraction: definitions/theorems from .tex → obj/mor
% - Coq/Isabelle adapters: demonstrate language agnosticism
%
% Analysis plugins (direction: graph → metadata → visual properties):
% - Centrality analysis: degree, betweenness, PageRank → node size
% - Community detection: Louvain, spectral clustering → node color groups
% - Path analysis: critical paths, bottleneck identification → highlight annotations
% - Custom analysis: user/domain-specific algorithms
%
% Why analysis plugins are not hardcoded in core: different domains and scales need different analysis methods. Core stays minimal — only obj/mor CRUD, rendering, sort system — analysis entirely by plugins.
