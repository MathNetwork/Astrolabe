\section{Foundations}
\label{sec:foundations}

We build the theory from a single starting point: a finite alphabet. Every externalized piece of knowledge is a finite string; structured knowledge is a finite sequence of key-value pairs of strings. From this minimal assumption we construct hash identifiers, semantic embeddings, and a reference structure, arriving at the \emph{astrolabe structure} $\mathcal{S}(\Sigma)$. The astrolabe structure is then graded by reference count into a \emph{semantic simplicial complex} $\Delta(\Sigma)$, on which the algebraic topology of \S\ref{sec:topology} operates.

\subsection{Alphabet and String Space}

\begin{definition}[Alphabet]
\label{def:alphabet}
An \emph{alphabet} is a finite set $A$. We impose no structure on $A$ beyond finiteness; it may be the set of Unicode code points, ASCII characters, or any other finite encoding.
\end{definition}

\begin{definition}[Monoid]
\label{def:monoid}
A \emph{monoid} is a triple $(M, \cdot, e)$ where $M$ is a set, $\cdot : M \times M \to M$ is an associative binary operation, and $e \in M$ is an identity element: $e \cdot m = m \cdot e = m$ for all $m \in M$.
\end{definition}

\begin{definition}[String space]
\label{def:string-space}
The \emph{string space} $A^*$ is the free monoid on $A$: the set of all finite sequences of elements of $A$, with concatenation as the monoid operation and the empty sequence $\varepsilon$ as identity.
\end{definition}

\begin{example}
Let $A = \{\texttt{a}, \ldots, \texttt{z},\; \text{`\textvisiblespace'}\}$, where `\textvisiblespace' denotes the space character, itself an element of the alphabet with no special status. The strings ``\texttt{compact}'' and ``\texttt{\textvisiblespace set}'' are elements of $A^*$. Their concatenation is ``\texttt{compact\textvisiblespace set}'' $\in A^*$. This operation is associative: concatenating ``\texttt{every\textvisiblespace}''$\cdot$``\texttt{compact\textvisiblespace}''$\cdot$``\texttt{set}'' in any grouping yields ``\texttt{every\textvisiblespace compact\textvisiblespace set}''. The identity element is the \emph{empty string} $\varepsilon$ (a sequence of length zero), not the space character `\textvisiblespace' (which is a sequence of length one): $\varepsilon \cdot s = s \cdot \varepsilon = s$ for all $s \in A^*$, whereas $\text{`\textvisiblespace'} \cdot s \neq s$ in general. These are precisely the monoid axioms. The qualifier \emph{free} means that $A^*$ has no relations beyond associativity and unit: distinct sequences of characters are distinct elements.
\end{example}

Every \LaTeX\ file, every Lean source, every JSON document, and every natural-language paragraph is an element of $A^*$ for an appropriate choice of $A$.

\begin{example}
Two natural choices of alphabet:
\begin{itemize}
\item $A_{\mathrm{ASCII}} = \{$ \texttt{a}, \texttt{b}, \ldots, \texttt{z}, \texttt{A}, \ldots, \texttt{Z}, \texttt{0}, \ldots, \texttt{9}, \texttt{+}, \texttt{-}, \texttt{/}, \texttt{=}, \ldots $\}$, $|A_{\mathrm{ASCII}}| = 128$. A Lean~4 source file is an element of $A_{\mathrm{ASCII}}^*$.
\item $A_{\mathrm{Unicode}}$, the set of all Unicode code points, $|A_{\mathrm{Unicode}}| = 149{,}813$ (as of Unicode 16.0). A Chinese mathematical textbook, a legal statute in Arabic, and a Lean file with Greek-letter variable names are all elements of $A_{\mathrm{Unicode}}^*$.
\end{itemize}
The choice of $A$ does not affect the theory; it only determines which strings are representable.
\end{example}

The free monoid is the most general algebraic structure that respects this observation.

\begin{theorem}[Universal property of $A^*$]
\label{thm:free-monoid}
For any monoid $(M, \cdot, e)$ and any function $f : A \to M$, there exists a unique monoid homomorphism $\bar{f} : A^* \to M$ extending $f$.
\end{theorem}

\begin{proof}
Define $\bar{f}(\varepsilon) = e$ and $\bar{f}(a_1 a_2 \cdots a_n) = f(a_1) \cdot f(a_2) \cdots f(a_n)$. This is a monoid homomorphism: $\bar{f}(uv) = \bar{f}(u) \cdot \bar{f}(v)$ by associativity of $\cdot$. It extends $f$: for $a \in A$, $\bar{f}(a) = f(a)$. Uniqueness: any homomorphism $g : A^* \to M$ with $g|_A = f$ must satisfy $g(a_1 \cdots a_n) = g(a_1) \cdots g(a_n) = f(a_1) \cdots f(a_n) = \bar{f}(a_1 \cdots a_n)$.
\end{proof}

\begin{remark}
In Lean~4, the free monoid is \texttt{FreeMonoid} in Mathlib, and the universal property is \texttt{FreeMonoid.lift}. The theorem above is a corollary of the general adjunction $\mathrm{Free} \dashv U$ between the free monoid functor and the forgetful functor from monoids to sets.
\end{remark}

The universal property guarantees that any structure-preserving interpretation of the alphabet extends uniquely to an interpretation of all strings. This is the algebraic foundation of parsing: a parser is a monoid homomorphism from $A^*$ to a target monoid of structured representations.

\begin{example}
Consider the Lean~4 compiler as a function on individual characters: each character $a \in A$ is mapped to a token fragment. The universal property says this extends to a unique homomorphism $\bar{f} : A^* \to M$ where $M$ is the monoid of token sequences (under concatenation). In practice, $\bar{f}(\texttt{``theorem\textvisiblespace List.length\_append''})$ = $\bar{f}(\texttt{t}) \cdot \bar{f}(\texttt{h}) \cdots = [(\texttt{sort},\;\texttt{theorem}),\; (\texttt{declaration},\;\texttt{List.length\_append})]$. Similarly, an LLM embedding $h_{\mathrm{emb}}$ maps the same string to a vector in $\mathbb{R}^d$. Different homomorphisms extract different structures from the same string; the universal property ensures each is uniquely determined by its action on individual characters.
\end{example}

\begin{definition}[Record space]
\label{def:record-space}
A \emph{record} is a finite sequence of key-value pairs, where both keys and values are strings:
\[
  R = (A^* \times A^*)^*.
\]
A record $r = ((k_1, v_1), \ldots, (k_n, v_n)) \in R$ assigns to each key $k_i$ a value $v_i$. Keys may repeat; the order of pairs is significant.
\end{definition}

The token sequence from Example~2.8 is naturally a record:
\[
(\texttt{sort},\; \texttt{theorem}),\quad (\texttt{declaration},\; \texttt{List.length\_append}).
\]
Parsing, in this view, is a homomorphism $A^* \to R$ that transforms a flat string into a structured record.

\subsection{Astrolabe Structure}

Let $R = (A^* \times A^*)^*$ be the record space defined in \S\ref{def:record-space}. Records in isolation are unstructured. We now equip a finite subset of $R$ with three additional structures: identity, semantics, and reference.

\begin{definition}[Identity hash]
\label{def:identity-hash}
An \emph{identity hash} is an injective function $h_{\mathrm{id}} : R \to H$ from records to a set of identifiers $H$ (in practice, $H = \{0,1\}^{96}$, the set of 12-character hexadecimal strings). Two records are considered identical if and only if $h_{\mathrm{id}}(r_1) = h_{\mathrm{id}}(r_2)$.
\end{definition}

The identity hash is a content-addressing mechanism~\cite{Benet2014}: a record's identity is determined by its content, not by its location. This is the same principle underlying Git commits, IPFS~\cite{Benet2014}, and the Nix package manager~\cite{Dolstra2004}.

\begin{example}
In the reference implementation, $h_{\mathrm{id}}$ is computed in two ways depending on context:
\begin{itemize}
\item \emph{Import}: $h_{\mathrm{id}}(\texttt{``List.length\_append''}) = \texttt{SHA256}(\texttt{``List.length\_append''})[:12] = \texttt{b3057161fbc7}$. The hash is deterministic: re-importing the same Lean declaration always produces the same identifier.
\item \emph{Creation}: $h_{\mathrm{id}} = \texttt{UUID4}()[:12] = \texttt{79c3794ef407}$. When a user creates a new record interactively, a random identifier suffices because uniqueness, not reproducibility, is the requirement.
\end{itemize}
Both produce 12-character hexadecimal strings ($H = \{0,\ldots,9,\texttt{a},\ldots,\texttt{f}\}^{12}$, giving $16^{12} \approx 2.8 \times 10^{14}$ possible identifiers). A key property of cryptographic hashes~\cite{NIST2015} is \emph{content sensitivity}: changing a single character in the input produces an entirely different hash. For instance, $\texttt{SHA256}(\texttt{``List.length\_append''})[:12] = \texttt{b3057161fbc7}$, but $\texttt{SHA256}(\texttt{``List.length\_Append''})[:12] = \texttt{7be29eeb832a}$ (a completely unrelated value). This ensures that distinct records receive distinct identifiers with overwhelming probability.
\end{example}

\begin{definition}[Semantic hash]
\label{def:semantic-hash}
A \emph{semantic hash} is a function $h_{\mathrm{emb}} : R \to \mathbb{R}^d$ that maps each record to a point in a $d$-dimensional Euclidean space, such that semantically similar records are mapped to nearby points. The idea of representing meaning as a vector originates with Word2Vec~\cite{Mikolov2013} and has been extended to sentences~\cite{Reimers2019} and documents. In practice, $h_{\mathrm{emb}}$ is computed by a language model embedding (e.g., OpenAI \texttt{text-embedding-3}~\cite{OpenAIEmbeddings2024}, $d = 3072$).
\end{definition}

\begin{example}[Two faces of the $h_{\mathrm{id}}$/$h_{\mathrm{emb}}$ gap]
\label{ex:semantic-hash}
The two hashes can disagree in either direction.

\medskip
\noindent\textbf{Same meaning, different text} ($h_{\mathrm{id}}$ far, $h_{\mathrm{emb}}$ close). The Poincar\'{e} conjecture can be stated as ``every simply connected, closed $3$-manifold is homeomorphic to $S^3$'' or as ``a compact $3$-manifold with trivial fundamental group is a $3$-sphere.'' These are the same theorem (Perelman, 2003), but $h_{\mathrm{id}}$ assigns them unrelated hashes because every word differs.

This phenomenon has a precise counterpart in type theory~\cite{Barendregt1984}: $\beta$-reduction (applying a function), $\delta$-reduction (unfolding a definition), $\eta$-reduction (extensional equality of functions), and $\iota$-reduction (reducing pattern matches) all transform a term into a \emph{definitionally equal} term with different syntax. In Lean~4~\cite{deMoura2021}, the kernel considers $\beta\delta\eta\iota$-equivalent terms identical, but their string representations differ, so $h_{\mathrm{id}}$ distinguishes them. Homotopy type theory~\cite{HoTTBook} goes further: the univalence axiom identifies equivalent types with equal types, collapsing an even broader class of ``same meaning, different text'' pairs. The study of equality---what counts as ``the same''---is a central concern of both mathematics and knowledge representation; $h_{\mathrm{id}}$ and $h_{\mathrm{emb}}$ capture two extremes of this spectrum.

\medskip
\noindent\textbf{Same text, different meaning} ($h_{\mathrm{id}}$ close, $h_{\mathrm{emb}}$ close, yet mathematically distinct). Consider Pitts's regularity theorem for min-max minimal hypersurfaces in $M^{n+1}$:
\begin{center}
\begin{tabular}{ll}
$n \leq 6$: & smooth (Pitts, 1981; Schoen--Simon regularity). \\
$n \leq 7$: & \emph{false} (the Simons cone in $\mathbb{R}^8$ is a singular minimizer).
\end{tabular}
\end{center}
Changing a single digit ($6 \to 7$) turns a theorem into a false statement. Both $h_{\mathrm{id}}$ and $h_{\mathrm{emb}}$ see them as nearly identical; the proof dependency structure (which dimensional estimates are invoked) distinguishes them. This dimension sensitivity is pervasive in mathematics: the Poincar\'{e} conjecture itself requires Ricci flow for $n = 3$, the $h$-cobordism theorem for $n \geq 5$, and Casson handles for $n = 4$.
\end{example}

The identity hash captures \emph{textual} identity; the semantic hash captures \emph{meaning} similarity. Their interplay is a central theme of this paper (\S\ref{sec:semantics}).

\begin{definition}[Parsing]
\label{def:parsing}
A \emph{parsing procedure} is a function $P : A^* \to 2^R$ that extracts a set of records from a string.
\end{definition}

\begin{example}
The same Lean source file $s \in A^*$ can be parsed in different ways:
\begin{itemize}
\item $P_{\mathrm{compiler}}(s)$: the Lean~4 compiler reads \texttt{.ilean} artifacts and produces one record per declaration, with keys \texttt{name}, \texttt{sort}, \texttt{statement}, and edges recording type-dependencies.
\item $P_{\mathrm{LLM}}(s)$: a language model reads the same file as natural language and extracts records for each mathematical concept mentioned in comments and docstrings, with edges reflecting semantic relationships described in prose.
\item $P_{\mathrm{AST}}(s)$: a syntax-level parser extracts the abstract syntax tree, producing records for each \texttt{def}, \texttt{theorem}, \texttt{instance} node with edges following the syntactic nesting structure.
\end{itemize}
Each parser produces a different set of records from the same input string, and in general $P_{\mathrm{compiler}}(s) \neq P_{\mathrm{LLM}}(s) \neq P_{\mathrm{AST}}(s)$.
\end{example}

\begin{definition}[Reference]
\label{def:reference}
The \emph{reference function} $\operatorname{ref} : R \to 2^R$ maps each record to the set of records it references, read from key-value content.
\end{definition}

\begin{example}
A morphism record in the reference implementation has keys \texttt{source} and \texttt{target} whose values are identity hashes of other records:
{\scriptsize\begin{verbatim}
{"source": "b3057161fbc7", "target": "a1f9c3d2e8b0", "sort": "uses",
 "notes": "applies the inductive hypothesis to the tail of the list"}
\end{verbatim}}
\noindent Here $\operatorname{ref}(r) = \{r_{\texttt{b305...}},\; r_{\texttt{a1f9...}}\}$: the record references two other records via their identity hashes. The key \texttt{notes} carries the semantic content of the dependency---precisely the information that \LaTeX\ \texttt{\textbackslash ref} and blueprint \texttt{\textbackslash uses} lack.
\end{example}

\begin{definition}[Astrolabe structure]
\label{def:astrolabe-structure}
An \emph{astrolabe structure} is a quadruple
\[
  \mathcal{S}(\Sigma) = (R,\; h_{\mathrm{id}},\; h_{\mathrm{emb}},\; \operatorname{ref})
\]
where $R \subset (A^* \times A^*)^*$ is a finite set of records, $h_{\mathrm{id}} : R \to H$ is an injective identity hash, $h_{\mathrm{emb}} : R \to \mathbb{R}^d$ is a semantic hash, and $\operatorname{ref} : R \to 2^R$ is a reference function.
\end{definition}

\begin{definition}[Essential/auxiliary decomposition]
\label{def:essential-auxiliary}
Each record $r = ((k_1, v_1), \ldots, (k_n, v_n))$ decomposes into an \emph{essential part} $r_{\mathrm{ess}}$ and an \emph{auxiliary part} $r_{\mathrm{aux}}$. The identity hash depends only on the essential part: $h_{\mathrm{id}}(r) = h_{\mathrm{id}}(r_{\mathrm{ess}})$. The essential part contains the mathematical content (name, statement, proof); the auxiliary part contains metadata (timestamps, analysis results, rendering preferences).
\end{definition}

\begin{definition}[Enrichment]
\label{def:enrichment}
An \emph{enrichment} is a function $E : R \to R$ that modifies only the auxiliary part of each record, leaving the essential part (and hence the identity hash) unchanged. Each plugin in the reference implementation is an enrichment: the timestamp plugin writes \texttt{created\_at}; the network analysis plugin writes \texttt{pagerank}.
\end{definition}

\subsection{Semantic Simplicial Complex}

The reference function $\operatorname{ref}$ induces a grading on records by the number of records they reference. This grading is the entry point for simplicial methods.

\begin{definition}[$k$-semantic simplex]
\label{def:k-simplex}
A record $r \in R$ is a \emph{$k$-semantic simplex} if $|\operatorname{ref}(r)| = k + 1$. A $0$-simplex references one record (or none, if it is a standalone object); a $1$-simplex references two records (a morphism with source and target); a $2$-simplex references three records (a ternary relationship); and so on. Each simplex carries a hash $h_{\mathrm{id}}(r)$, key-value content, and a semantic coordinate $h_{\mathrm{emb}}(r) \in \mathbb{R}^d$.
\end{definition}

\begin{definition}[Semantic vacancy]
\label{def:vacancy}
A \emph{semantic vacancy} is a record whose content explicitly represents a gap in the knowledge base. Each type of vacancy has an independent hash and a sort indicating its nature:
\begin{itemize}
\item \emph{independent}: no relationship is expected.
\item \emph{missing}: a relationship is expected but not yet established.
\item \emph{induced}: a relationship follows from existing ones but has not been made explicit.
\item \emph{negated}: a relationship has been explicitly ruled out.
\end{itemize}
A vacancy is not the absence of a record; it is a record with specific content. This distinction is essential: the absence of an edge in a knowledge graph is ambiguous (not investigated? or investigated and found absent?), whereas a vacancy is unambiguous.
\end{definition}

\begin{definition}[Astrolabe complex]
\label{def:astrolabe-complex}
The \emph{astrolabe complex} is the graded set
\[
  \Delta(\Sigma) = \bigsqcup_{k \geq 0} \Delta_k
\]
where $\Delta_k$ is the set of all $k$-semantic simplices in $\mathcal{S}(\Sigma)$.
\end{definition}

\begin{definition}[Face maps]
\label{def:face-maps}
For each $0 \leq i \leq k$, the \emph{face map} $d_i : \Delta_k \to \Delta_{k-1}$ removes the $i$-th reference from a $k$-simplex. These maps satisfy the semi-simplicial identities
\[
  d_i \circ d_j = d_{j-1} \circ d_i \quad \text{for } i < j,
\]
making $\Delta(\Sigma)$ a semi-simplicial set.
\end{definition}

\begin{remark}
For $k = 1$, the face maps $d_0$ and $d_1$ recover the source and target maps of a directed graph: $d_1 = s$ (source) and $d_0 = t$ (target). The simplicial perspective generalizes the graph perspective to arbitrary dimension.
\end{remark}
