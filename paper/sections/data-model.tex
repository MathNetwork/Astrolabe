\section{Data Model}
\label{sec:data-model}

The core data structure of Astrolabe is a content-addressable store of \emph{entries}, where each entry is a triple of hash, ref, and record. Entries can have arbitrary-length references, forming higher-dimensional semantic structures. The store admits two orthogonal decompositions: by \emph{degree} (how many references) and by \emph{stage} (the depth of the reference chain). We describe each in turn.

\subsection{Entry Structure}

An \emph{astrolabe store} is a finite set of entries. Each entry consists of an ordered reference list pointing to other entries and an opaque record string, with its identity determined by the hash of its content.

\begin{definition}[Content-addressable hypergraph]
Let $H$ be a hash function. A \emph{content-addressable hypergraph} is a finite set $S$ of entries, where each entry $e \in S$ consists of:
\begin{itemize}
  \item an ordered reference list $\mathrm{ref}(e) = (r_1, \ldots, r_k)$ with $r_i \in S$, and
  \item a record string $\mathrm{rec}(e) \in \Sigma^*$,
\end{itemize}
with identity $\mathrm{id}(e) = H(\mathrm{id}(r_1) \| \texttt{0x00} \| \cdots \| \mathrm{id}(r_k) \| \mathrm{rec}(e))$. The \emph{degree} of $e$ is $|\mathrm{ref}(e)|$. An entry of degree~0 has $\mathrm{ref}(e) = (\mathrm{id}(e))$ and is called an \emph{atom}.
\end{definition}

The format is general-purpose and can serve any domain (mathematics, software, biology, legal, etc.) by defining appropriate record conventions. In the current implementation, the store is serialized as a single JSON file (\texttt{astrolabe.json}):

\medskip
{\small\ttfamily
\begin{tabular}{@{}l@{}}
\{\\
\ \ \textcolor{gray}{"<12-char-hash>"}:\{\textcolor{blue!70!black}{"ref"}:[\textcolor{gray}{"<hash>"}, ...], \textcolor{red!70!black}{"record"}:\textcolor{gray}{"<string>"}\}\\
\}
\end{tabular}
}
\medskip

\noindent Concretely, each entry has three parts:
\begin{itemize}
  \item \textbf{ref} --- an ordered list of hashes, any length. $|\texttt{ref}|$ defines the \emph{degree}:
    \begin{itemize}
      \item degree 0: \texttt{ref = [self\_hash]} --- an atom (base unit)
      \item degree 1: \texttt{ref = [A, B]} --- a binary relation
      \item degree $k$: \texttt{ref = [h$_0$, h$_1$, ..., h$_k$]} --- a higher-dimensional semantic relation
    \end{itemize}
  \item \textbf{record} --- a plain string. The core layer does not interpret it; plugins define conventions for structured content (JSON with \texttt{sort}, \texttt{source}, \texttt{title}, \texttt{notes}, etc.).
  \item \textbf{hash} --- $\texttt{SHA256}(\mathit{ref}_1 \| \texttt{0x00} \| \mathit{ref}_2 \| \cdots \| \mathit{record})\texttt{[:12 hex]}$, content-addressable.
\end{itemize}

\paragraph{Hash propagation.}
Because entries are content-addressed, modifying a record changes its hash. When this happens, every entry that references the old hash must be updated and re-hashed recursively. Figure~\ref{fig:propagation} illustrates: editing the record of atom \texttt{a} changes its hash from \texttt{a} to \texttt{a\textsuperscript{$\prime$}}; the edge \texttt{e1} (which references \texttt{a}) is updated to reference \texttt{a\textsuperscript{$\prime$}} and re-hashed to \texttt{e1\textsuperscript{$\prime$}}; then \texttt{e3} (which references \texttt{e1}) is updated and re-hashed to \texttt{e3\textsuperscript{$\prime$}}.

\subsection{Degree Decomposition}

Figure~\ref{fig:astrolabe-network} shows a 16-entry astrolabe file and its reference network. The entries span multiple degrees: 4~atoms (\texttt{a1}--\texttt{a4}), 4~degree-1 edges (\texttt{e1}--\texttt{e4}), and higher-degree entries that freely mix atoms and edges in their \texttt{ref} lists. The entries \texttt{c1}, \texttt{c2}, \texttt{c3} form a reference cycle. The entry \texttt{m2} references \texttt{m1} (which itself references \texttt{f1}, a degree-2 entry), demonstrating that references are unconstrained: any entry can reference any other entry, regardless of degree.

\subsection{Stage Decomposition}

Degree measures how many references an entry has, but not \emph{what those references point to}. Two degree-1 entries can have very different structure: \texttt{e1} references two atoms, while \texttt{f2} references two edges. The \emph{stage} decomposition captures this depth. Stages are defined inductively: $\mathcal{S}_0$ = atoms; $\mathcal{S}_{m+1}$ = entries whose refs all belong to $\mathcal{S}_m$. Cyclic entries belong to no finite stage.

Figure~\ref{fig:stage-coloring} shows the same network colored by stage instead of degree. The key contrast: \texttt{e1} and \texttt{f2} both have degree~1, but \texttt{e1} is stage~1 (refs are atoms) while \texttt{f2} is stage~2 (refs include stage-1 entries). The entry \texttt{m1} is stage~3 (references \texttt{f1} from stage~2), and \texttt{m2} is stage~4. Within each stage layer, every reference draws from the same well-defined hash pool, enabling consistent structural analysis.
