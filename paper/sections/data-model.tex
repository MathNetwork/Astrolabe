\section{The Categorical Data Model}
\label{sec:data-model}


This section presents the formal data model underlying Astrolabe. We define the \emph{Astrolabe category}---a free category generated by a labeled directed graph---and describe its sort system, identifier design, and core vocabularies.


\subsection{The Knowledge Graph}
\label{sec:dm-knowledge-graph}

The knowledge graph is the data layer of the system---the persistent representation of all objects, morphisms, sorts, and identifiers that a user or plugin has created. In the implementation, it corresponds directly to the \texttt{knowledge.json} file and the CRUD API that reads and writes it.

\begin{definition}[Astrolabe Knowledge Graph]
\label{def:knowledge-graph}
An \textbf{Astrolabe knowledge graph} is a labeled directed graph $G = (O, M, s, t, \sigma_O, \sigma_M, h)$ where:
\begin{itemize}
    \item $O$ is a set of \emph{objects} (knowledge units: definitions, theorems, lemmas, conjectures, \ldots),
    \item $M$ is a set of \emph{morphisms} (typed relationships between objects),
    \item $s, t: M \to O$ are source and target maps,
    \item $\sigma_O: O \to S_O$ and $\sigma_M: M \to S_M$ are \emph{sort} classification functions mapping objects and morphisms to their respective sort label sets,
    \item $h: O \cup M \to \mathrm{Hex}_{12}$ assigns a unique 12-character hexadecimal identifier to each entity.
\end{itemize}
Each object $o \in O$ carries optional rich-text fields: \texttt{name}, \texttt{statement}, \texttt{proof}, \texttt{intuition}, \texttt{notes} (in MDX format), and an extensible \texttt{metadata} dictionary. Each morphism $m \in M$ carries optional \texttt{sort}, \texttt{notes}, and \texttt{metadata} fields.
\end{definition}

Objects and morphisms share the identifier space $\mathrm{Hex}_{12}$ and are stored as parallel dictionaries in a single JSON file (\texttt{knowledge.json}). This design makes morphisms first-class citizens at the storage level: they have the same identifier format, the same metadata extensibility, and the same CRUD interface as objects. \subsection{The Free Category}
\label{sec:dm-free-category}

The knowledge graph $G$ stores only direct relationships---the morphisms that users and plugins explicitly create. But knowledge has transitive structure: if Theorem~A uses Lemma~B and Lemma~B uses Definition~C, then A transitively depends on C. The free category generated by $G$ captures this transitive closure while keeping the stored representation compact. In the system, the free category corresponds to the \emph{framework}: the core runtime that maintains the graph, computes compositions on demand (e.g., path queries, reachability analysis), and provides the substrate on which plugins operate.

\begin{definition}[Astrolabe Category]
\label{def:astrolabe-category}
An Astrolabe knowledge graph $G$ freely generates a category $\mathcal{A}(G)$, the \textbf{Astrolabe category}:
\begin{itemize}
    \item The objects of $\mathcal{A}(G)$ are the elements of $O$.
    \item The morphisms of $\mathcal{A}(G)$ are finite composable paths of elements of $M$, together with an identity morphism $\mathrm{id}_o$ for each $o \in O$.
    \item Composition is concatenation of paths: for $f: A \to B$ and $g: B \to C$, the composite $g \circ f: A \to C$ is the path obtained by appending $g$ to $f$.
    \item Composition is associative, and identity morphisms are neutral elements.
\end{itemize}
\end{definition}

$\mathcal{A}(G)$ is the \emph{free category} on the graph $G$. The morphisms in $M$ are the generators; composite morphisms are not stored explicitly but computed on demand via graph traversal. This is a deliberate design choice: storing all compositions would cause combinatorial explosion ($n$ composable morphisms yield $O(2^n)$ paths), while the free category's generator representation is compact. Figure~\ref{fig:free-category} illustrates the distinction.

\subsection{Import Functors and Analysis Enrichments}
\label{sec:dm-functors}

The Astrolabe category $\mathcal{A}(G)$ defines \emph{what} the knowledge graph is; functors and enrichments define \emph{how external data enters it} and \emph{how derived information is computed on it}. In the system, these correspond to the plugin architecture: import plugins bring data from external sources (Lean compilation artifacts, BibTeX files, other knowledge bases) into the graph as standard obj/mor, and analysis plugins compute network metrics that attach to existing entities as metadata. The categorical formalization ensures that plugins interact with the graph through a well-defined interface rather than ad-hoc data manipulation.

\begin{definition}[Astrolabe Plugin Functor]
\label{def:plugin-functor}
An \textbf{Astrolabe plugin} is a functor $F: \mathcal{C} \to \mathcal{A}(G)$ from a source category $\mathcal{C}$ to the Astrolabe category. $F$ maps objects of $\mathcal{C}$ to objects of $\mathcal{A}(G)$ and morphisms of $\mathcal{C}$ to morphisms of $\mathcal{A}(G)$, preserving composition and identities. The output of $F$ is standard obj/mor---subject to the same sort system, identifier space, and rendering pipeline as manually created entities.
\end{definition}

Two classes of plugins arise as instances of this definition.

\begin{example}[Import Functor: $F_{\mathrm{Lean}}$]
\label{ex:import-functor}
The ilean parser implements $F_{\mathrm{Lean}}: \mathcal{L} \to \mathcal{A}(G)$, where $\mathcal{L}$ is the category whose objects are Lean declarations and whose morphisms are type dependencies. Figure~\ref{fig:import-functor} illustrates the mapping on a concrete fragment.

\subsection{Sort System}
\label{sec:dm-sort-system}

The sort labels $S_O$ and $S_M$ are free strings, not fixed enumerations. Any string is a valid sort; the system imposes no closed vocabulary. This is the key to domain agnosticism: a mathematics project uses sorts like \texttt{theorem}, \texttt{definition}, \texttt{uses}; a legal project might use \texttt{statute}, \texttt{case}, \texttt{interprets}; a biology project might use \texttt{gene}, \texttt{pathway}, \texttt{regulates}.

A set of \emph{core sorts} receives system-level treatment---special rendering rules, analysis weights, or status tracking. Table~\ref{tab:obj-sorts} lists the core object sorts; the full morphism sort vocabulary (12 types across four categories: structural, logical, semantic, and meta) is detailed in Appendix~\ref{sec:app-mor-vocab}. The four most important morphism sorts are:

\begin{itemize}
    \item \textbf{uses} ($u: A \to B$): $A$'s statement logically depends on $B$. Rendered as a solid arrow.
    \item \textbf{proof-uses} ($p: A \to B$): $A$'s proof uses $B$, but $A$'s statement does not depend on $B$. Rendered as a dashed arrow. The distinction matters for network analysis: proof-uses edges should carry lower weight than uses edges in centrality computations, since they represent technique choices rather than logical necessity.
    \item \textbf{inclusion} ($i: A \to B$): $A$ belongs to $B$ (part-of relationship). Transitive; forms a DAG. Rendered as nesting or clustering. Inclusion subsumes hierarchical organization---chapters containing sections containing definitions---without a separate folder/tree mechanism.
    \item \textbf{formalizes} ($f: A \to B$): $A$ (a formal declaration) is a machine-verified implementation of $B$ (a natural-language knowledge unit). Supports many-to-many: one theorem may be formalized as multiple Lean declarations, and one Lean declaration may participate in formalizing multiple theorems.
\end{itemize}

Sorts outside the core vocabulary automatically receive a deterministic color via an FNV-1a hash mapped to the HSL color wheel. This ensures that any new sort---whether user-defined or plugin-provided---is immediately visually distinguishable without configuration.

\subsection{Identifier Design}
\label{sec:dm-identifiers}

Identifiers operate at two layers:

\begin{enumerate}
    \item \textbf{Core layer.} Each object and morphism receives a random 12-character hexadecimal identifier at creation time. These IDs are stable across edits---changing a theorem's statement does not change its identity---and serve as the primary key for all internal references (MDX cross-links, morphism source/target, analysis metadata).

    \item \textbf{Formalization layer.} The import functor $F_{\mathrm{Lean}}$ computes identifiers deterministically: $h = \mathrm{SHA256}(\textit{lean\_full\_name})[:12]$. This ensures that the same Lean declaration always maps to the same Astrolabe ID, enabling idempotent re-import and cross-instance deduplication. When content-addressed hashes from type signatures are available (following the pipeline described in~\cite{LiParisSeverini2026}), they can be attached as metadata, enabling merging across independently maintained Astrolabe instances.
\end{enumerate}

The two-layer design reflects a fundamental asymmetry: natural-language content has no canonical form (rephrasing a definition changes its text but not its identity), while formal content does (two Lean declarations with identical type signatures represent the same mathematical object regardless of naming).

\paragraph{Canonicalization as quotient category.} The content-addressing pipeline~\cite{LiParisSeverini2026} defines two canonicalization levels for Lean expressions: L0 (universe parameter renaming + metadata stripping, pure) and L1 (reducible-transparency normalization via \texttt{whnf} at all subexpressions, then L0). Each level $\ell$ induces an equivalence relation $\sim_\ell$ on type expressions. Hashing at level $\ell$ effectively computes in the quotient category $\mathcal{A}(G)/{\sim_\ell}$: objects whose type expressions are $\ell$-equivalent receive the same content hash and are identified as the same knowledge unit. L0 identifies declarations differing only in naming conventions; L1 additionally identifies declarations differing by reducible unfoldings. This provides a formal mechanism for controlling the granularity of deduplication---a stricter level produces larger equivalence classes and more aggressive merging.


\subsection{MDX as Content Substrate}
\label{sec:dm-mdx}

Astrolabe uses MDX (Markdown with JSX) rather than \LaTeX{} as its content format. The choice is driven by three requirements that \LaTeX{} does not satisfy.

First, MDX content is \emph{structurally addressable}. An \texttt{<ObjBlock id="3965488135e9"/>} tag in an MDX document is a live reference to an object in the knowledge graph---it renders the object's current content, responds to clicks by navigating to the object, and updates when the object changes. \LaTeX{}'s \texttt{\textbackslash ref} mechanism references labels, not first-class entities with typed relationships.

Second, MDX supports \emph{interactive components}. React components embedded in MDX can render geometric objects with rotation and zoom, animate variational processes, or provide collapsible proof steps. \LaTeX{} output is static PDF---a closed terminal format that cannot be programmatically consumed or interacted with. The executable papers paradigm~\cite{Koop2011} pursued a similar goal of making research content interactive; Astrolabe achieves this by making interactive components part of the knowledge graph rather than attachments to a linear document.

Third, MDX is \emph{AI-compatible}. Large language models generate valid MDX more reliably than structured \LaTeX{} (particularly TikZ diagrams), and MDX components can be generated on demand---a user requesting ``show the proof tree for this theorem'' receives an \texttt{<ProofTree root="abc123"/>} component from the AI, rendered immediately in the document.

Mathematical notation is preserved: MDX embeds \LaTeX{} formulas via KaTeX, so display equations, inline symbols, and theorem environments render with full typographic quality. What MDX replaces is \LaTeX{}'s \emph{document framework}---the linear, file-based, compilation-dependent structure---not its formula typesetting.


\subsection{The Closed-Loop Pipeline}
\label{sec:dm-pipeline}

The categorical data model enables a closed-loop pipeline for mathematical formalization, illustrated in Figure~\ref{fig:pipeline-loop}. Users structure natural-language content as obj/mor with AI assistance (Phase~1). Analysis plugins compute centrality, community, and bottleneck signals on the typed network (Phase~2). These signals guide which nodes to formalize, whether by human effort or AI-assisted tools (Phase~3). Formalization artifacts (\texttt{.ilean} files) flow back through the import functor $F_{\mathrm{Lean}}$ as standard obj/mor (Phase~4). The network updates, analysis re-runs, and the cycle repeats.

The pipeline works because $F_{\mathrm{Lean}}$ produces obj/mor that are indistinguishable from manually created entities. A \texttt{lean-theorem} object immediately participates in morphisms with existing natural-language objects, inherits the sort color system, and appears in network analysis results. No format conversion or adapter layer is needed.


\subsection{Three-Layer Coexistence}
\label{sec:dm-coexistence}

When formalization artifacts flow back into the graph, natural-language objects and formal objects coexist. This coexistence operates at three layers.

\paragraph{Data model layer: many-to-many mapping.} A single natural-language theorem may be formalized as multiple Lean declarations (a main statement, auxiliary lemmas, helper constructions), each connected via \texttt{formalizes} morphisms. Conversely, a single Lean lemma may participate in formalizing multiple theorems. The typed morphism model supports this natively---many-to-many relationships are just multiple morphisms, with no special mechanism required.

\paragraph{Rendering layer: sort-driven visual differentiation.} Different morphism sorts trigger different visual representations on the Canvas: \texttt{uses} as solid arrows, \texttt{proof-uses} as dashed arrows, \texttt{formalizes} as double-width colored lines, \texttt{contradicts} as red lines. The same underlying obj/mor data model drives all renderings; the sort value alone determines the visual mode. Rendering rules are defined in a registry that plugins can extend.

\paragraph{Content layer: MDX component DSL.} MDX components form a domain-specific language for referencing knowledge graph entities. Basic components (\texttt{<ObjBlock>}, \texttt{<ObjRef>}) provide live references to objects. Composite components (\texttt{<NodeCluster>} for inclusion subtrees, \texttt{<FormalizationStatus>} for coverage metrics) compose from basic components and graph queries. The AI can generate these components on demand, acting as a code generator for the DSL.
