\section{Introduction}
\label{sec:introduction}


Structured knowledge---mathematical theories, legal codes, biological pathways, software architectures---is inherently networked: concepts depend on other concepts, results build on prior results, entities relate to each other in typed, directional ways. Yet the software we use to manage this knowledge treats it as a collection of documents with untyped hyperlinks. This paper argues that the basic software primitives for knowledge management should be category-theoretic \emph{objects} and \emph{morphisms}, and presents Astrolabe, a system that implements this principle.


\subsection{The Problem: Second-Class Citizenship of Relations}
\label{sec:intro-problem}

Nearly every knowledge management system in widespread use today---Obsidian, Notion, Logseq, MediaWiki, Roam Research---treats relationships between knowledge units as second-class citizens. A link from page~A to page~B is a bare pointer: it has no independent identity, no type, no content of its own. The link exists only as a property of~A, not as a first-class entity that can itself be referenced, searched, annotated, or analyzed.

In any structured knowledge domain, relationships carry information as critical as the entities they connect. In mathematics, ``Theorem~A \emph{implies} Theorem~B'' and ``Theorem~A \emph{uses} Theorem~B'' are semantically distinct---the first asserts a logical consequence, the second declares a proof-technique dependency---yet existing tools render both as the same unlabeled line. In law, the relationship between a statute and a case that interprets it differs in kind from the relationship between two cases that contradict each other. In biology, ``gene~X \emph{regulates} gene~Y'' and ``gene~X \emph{is homologous to} gene~Y'' demand different analytical treatment. Current tools collapse all of these into a single, undifferentiated edge type.

The consequences run deeper than visual impoverishment. When relationships lack independent identity, they cannot be referenced from other documents (``see the dependency between Theorem~3.2 and Lemma~2.7''). When they lack types, network analysis algorithms---centrality, community detection, bottleneck identification---produce signals that conflate logically distinct dimensions. A node with high betweenness centrality in a graph where all edges mean ``is somehow related to'' tells us far less than one where edges are typed as ``depends on for its statement'' versus ``uses in its proof.'' The absence of typed edges is an architectural barrier to treating knowledge networks as objects of computational analysis.


\subsection{From Ologs to Astrolabe}
\label{sec:intro-ologs}

The theoretical foundations for making relations first-class exist: Spivak and Kent's \emph{Ologs}~\cite{Spivak2012} showed that category theory provides a rigorous framework for knowledge representation, with objects as types and morphisms as typed relations. But ologs never produced a practical tool---the formalism's barrier proved prohibitive, and its restriction to functional relations is too rigid for knowledge management (Section~\ref{sec:bg-category-theory}). This paper extends the olog foundation in three directions: from theory to a runnable system (with AI to lower the barrier), from functional relations to an open \emph{sort} vocabulary (enabling one-to-many and soft associations), and into engineering dimensions ologs did not address (content-addressing, network analysis, plugins).


\subsection{Motivation: The Blueprint--Autoformalization Gap}
\label{sec:intro-gap}

Mathematical formalization provides the sharpest application scenario. Blueprint tools like leanblueprint~\cite{Massot2020} enable dependency planning; autoformalization tools produce increasing volumes of verified code. Between these two ends lies a structural gap: no tool provides a unified knowledge layer where natural-language mathematics and machine-verified formalizations coexist as typed entities in the same network, with network analysis guiding formalization strategy (Section~\ref{sec:bg-blueprints}).


\subsection{Contributions}
\label{sec:intro-contributions}

This paper makes four contributions:

\begin{enumerate}
    \item \textbf{Software primitives.} We propose category-theoretic \emph{objects} and \emph{morphisms} as the basic software primitives for knowledge management. Both hold equal status: each possesses an independent unique identifier, a typed label (sort), and rich-text content. We rigorously define the \emph{Astrolabe category}---a free category generated by a knowledge graph, equipped with 12~core morphism types and an open sort extension mechanism (Section~\ref{sec:data-model}).

    \item \textbf{Closed-loop pipeline.} We demonstrate a network-analysis-driven pipeline for mathematical formalization (Figure~\ref{fig:pipeline-loop}, Section~\ref{sec:dm-pipeline}).

    \item \textbf{System implementation.} We present Astrolabe, a desktop knowledge graph framework implementing this model. Its architecture follows a minimal-core-plus-plugins design where import plugins are functors, analysis plugins are enrichments, and AI acts as a proposal generator for the knowledge reference DSL (Section~\ref{sec:system}).

    \item \textbf{Empirical validation.} We evaluate the model by importing Mathlib's dependency network (100k+ declarations) into Astrolabe's obj/mor format and comparing the analytical capabilities enabled by typed morphisms against existing flat-graph tools (Section~\ref{sec:evaluation}).
\end{enumerate}
