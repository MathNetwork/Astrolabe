\section{Introduction}
\label{sec:introduction}

Knowledge management tools broadly fall into two categories: document-centric systems (wikis, note-taking apps) that store prose but lose structural relationships, and graph databases that capture relationships but lack accessible visualization and authoring interfaces.

Note-taking tools such as Obsidian~\cite{Obsidian}, Roam Research~\cite{RoamResearch}, and Logseq~\cite{Logseq} introduced bidirectional links, turning notes into graphs. However, these links carry no semantic content: \texttt{[[page name]]} records \emph{that} a connection exists but not \emph{what kind}. Knowledge graphs in the RDF/OWL tradition provide typed edges, but the type vocabulary is a fixed enumeration --- adding a new relationship type requires schema modification. Content-addressable systems such as Git and IPFS/IPLD~\cite{IPLD} provide immutable, hash-identified objects, but impose fixed object types and do not support higher-dimensional semantic structures.

On the theoretical side, Spivak and Kent~\cite{Spivak2012} proposed \emph{ologs} (ontology logs) as a categorical framework for knowledge representation: objects are types, morphisms are functional relations, and the category-theoretic structure enforces consistency (Figure~\ref{fig:olog}). This was a significant conceptual advance. However, ologs restrict morphisms to functional relations (every object maps to exactly one target) and cannot carry open-ended content. Moreover, the framework has seen limited software adoption since its introduction.

On the engineering side, the closest existing data model is HyperGraphDB~\cite{HyperGraphDB}, a generalized graph database where every entity is an \emph{atom} with a target set (a list of references to other atoms). Atoms with an empty target set are pure nodes; atoms with a non-empty target set are hyperedges --- and since hyperedges are themselves atoms, they can be referenced by other hyperedges, yielding higher-order relationships (Figure~\ref{fig:hgdb}).

Astrolabe adopts the same core abstraction --- every entry has a \texttt{ref} list pointing to other entries --- but adds two properties that HyperGraphDB lacks:
\begin{enumerate}
  \item \textbf{Content-addressable identity}: entries are identified by the SHA-256 hash of their content (as in Git/IPFS~\cite{IPLD}), providing deduplication, tamper evidence, and decentralized identity.
  \item \textbf{Automatic hash propagation}: when an entry is modified, its hash changes, and all referencing entries are recursively updated, maintaining referential integrity across the entire network.
\end{enumerate}

Astrolabe intentionally adopts this \emph{maximally relaxed} data structure to accommodate diverse and complex semantics. In \S\ref{sec:data-model} we define the data format \texttt{astrolabe.json} and describe two orthogonal decompositions: by \emph{degree} (the number of references an entry has) and by \emph{stage} (the depth of the reference chain). In \S\ref{sec:plugins} we present the MathNetwork plugin that bridges informal and formal mathematics. \S\ref{sec:visualization} describes the interactive visualization, and \S\ref{sec:architecture} summarizes the system architecture.
