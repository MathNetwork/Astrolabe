# Astrolabe Category

**A Categorical Foundation for Knowledge Management**

Astrolabe category is a free category generated by a signature $\Sigma = (O, M, h)$, where objects and morphisms carry open information records and share a global identifier space.

**This is an astrolabe category.** `backend/` is the signature $(O, M, h)$ and its functors, `formal/` is the specification that constrains them, `paper/` is the theory that defines them.

## Structure

```
astrolabe-category/
├── backend/
│   └── astrolabe/
│       ├── signature/              # (O, M, h)
│       │   ├── obj.py              #   O — objects
│       │   ├── mor.py              #   M — morphisms
│       │   └── state.py            #   runtime state
│       ├── functors/
│       │   ├── builtin/lean/       #   F_Lean : Lean → A(Σ)
│       │   └── analysis/           #   A : A_M → A_M'
│       ├── knowledge_storage.py    # h — persistence
│       └── server.py               # API
├── src/                            # functors to V, U_ui, P (Next.js)
│   ├── panels/                     #   P : A(Σ) → U_ui
│   ├── lib/graph2d.ts              #   L : A(Σ) → P (d3-force)
│   ├── lib/sortConfig.ts           #   R : A(Σ) → V (color/size)
│   ├── stores/                     #   state management (zustand)
│   └── hooks/                      #   data loading, shortcuts
├── paper/                          # narrative (LaTeX)
├── formal/                         # specification (Lean 4, planned)

```

`backend/astrolabe/signature/` is $(O, M, h)$. `backend/astrolabe/functors/` contains functors whose target is $\mathcal{A}(\Sigma)$ itself (import, analysis). `src/` contains functors whose target is external ($\mathcal{V}$, $\mathcal{U}_{\text{ui}}$, $\mathcal{P}$) — its directory structure follows Next.js conventions. `formal/` is the specification. `paper/` is the theory.

## Astrolabe Category

An **astrolabe signature** $\Sigma = (O, M, h)$ consists of:

- $O$ — objects, each carrying an open information record
- $M$ — morphisms (first-class), each carrying source, target, and information record
- $h : O + M \to \text{Hex}_{12}$ — injective global identifier

The **astrolabe category** $\mathcal{A}(\Sigma) := \text{Free}(\Sigma)$ is the free category generated by $\Sigma$.

All operations on $\mathcal{A}(\Sigma)$ are **functors**:

| Functor | Signature | Instance |
|---------|-----------|----------|
| Import | $F: \mathcal{E} \to \mathcal{A}(\Sigma)$ | $F_{\text{Lean}}$, $F_{\text{BibTeX}}$ |
| Analysis | $A: \mathcal{A}_{\mathcal{M}} \to \mathcal{A}_{\mathcal{M}'}$ | centrality, community detection |
| Rendering | $R: \mathcal{A}(\Sigma) \to \mathcal{V}$ | autoColor, core sort rules |
| Presentation | $P: \mathcal{A}(\Sigma) \to \mathcal{U}_{\text{ui}}$ | MDX components |
| Layout | $L: \mathcal{A}(\Sigma) \to \mathcal{P}$ | d3-force |

## Formal Verification

`formal/` contains Lean 4 proofs that serve as **specification** in two senses:

- **Algebraic specification**: astrolabe signature + axioms define the category (cf. Goguen & Burstall 1992)
- **Software specification**: Lean proofs constrain the Python implementation

Two layers of verification:

```
formal/     specification — what the code must satisfy (Lean 4)
tests/      engineering   — does the code run correctly (pytest + jest)
```

They do not replace each other. `formal/` verifies mathematical properties. `tests/` verifies engineering properties.

The formalization is imported back into astrolabe via $F_{\text{Lean}}$ — astrolabe manages the formal proofs of its own theory.

```bash
# frontend
npm install && npm run build

# backend
cd backend && pip install -r requirements.txt
python -m astrolabe.server

# formal verification
cd formal && lake build

# tests
npm test
cd backend && python -m pytest
```
