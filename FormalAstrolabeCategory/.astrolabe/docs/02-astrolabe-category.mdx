## 2.6 Split Functors and Functor Pipelines

**Definition (Split functor).** A *split functor* $F: \mathcal{A}(\Sigma) \to \mathcal{T}$ is a triple $(F_b, \alpha, F_f)$ where:
- $F_b: \mathcal{A}(\Sigma) \to \mathcal{D}$ is the *backend component* (computation)
- $F_f: \mathcal{D} \to \mathcal{T}$ is the *frontend component* (rendering)
- $\alpha: F_b \to F_f$ is the *API bridge* — a natural transformation that transmits computed data to the rendering layer

such that $F = F_f \circ \alpha \circ F_b$. The API endpoints are the components of $\alpha$.

**Example.** The network analysis functor splits as:
- $F_b$: computes pagerank, betweenness from the morphism graph (Python)
- $\alpha$: REST API `/api/project/analysis/*` (JSON serialization)
- $F_f$: maps pagerank values to node sizes on canvas (TypeScript)

**Definition (Functor pipeline).** A *functor pipeline* is a composable chain of enrichment functors applied during object creation:

$$\text{create} = E_n \circ \cdots \circ E_2 \circ E_1 \circ h$$

where $h$ generates the identifier and each $E_i: \mathcal{A}(\Sigma) \to \mathcal{A}(\Sigma)$ is an enrichment functor. The pipeline is:

$$h \xrightarrow{E_{\text{math}}} \text{defaults} \xrightarrow{E_{\text{time}}} \text{timestamps} \xrightarrow{\text{user}} \text{override}$$

**Proposition (Pipeline commutativity).** If two enrichment functors $E_i, E_j$ write to disjoint keys, then $E_i \circ E_j = E_j \circ E_i$. (Proved in Lean: `timestamp_mathDomain_commute`.)

**Definition (Functor registry).** The *functor registry* $\mathcal{R}$ is the coproduct of all active functor frontend components:

$$\mathcal{R} = \coprod_{i} F_{f,i}$$

A panel queries $\mathcal{R}$ to obtain the appropriate rendering component for each object. This makes panels *thin containers* — they hold no domain logic, only layout.

**Proposition (Inter-functor communication).** Communication between functors $F$ and $G$ on the same side (both frontend or both backend) is a natural transformation $\eta: F \Rightarrow G$. The naturality condition ensures that the communication is coherent across all objects:

$$\eta_B \circ F(f) = G(f) \circ \eta_A \quad \text{for all } f: A \to B$$

**Example.** The rendering functor $R$ reads the network analysis functor $A$'s output to determine node size. This is $\eta: A \Rightarrow R$ where $\eta_o$ maps the analysis result at object $o$ to the visual size at object $o$, and naturality ensures that connected objects have consistently scaled sizes.
