# Derived Numbering

A card's identity is its hash; a number like "Theorem 7.2.8" is how a *project*
presents that card to a reader. Numbers are therefore **derived,
never stored**: a pure positional coordinate computed from the project's
documents alone, assigned fresh on every load. There is no number field anywhere
in the store, and none to ever go stale.

## The coordinate

Each card's number is `chapter.section.item`, read off the mdx documents:

- **chapter** — the document's `# Chapter N` title (or its filename prefix);
- **section** — the ordinal of the `##` heading the card sits under: the first
  `##` heading opens section 1, the second section 2, and so on. Headings do not
  need numbers in their text; content before the first heading is section 0.
- **item** — the card's order among the section's `\entryblock`s.

## The three rules

- **First occurrence wins.** Documents are scanned in filename order, and a hash
  is numbered where its `\entryblock` first appears; every later occurrence —
  block or reference — reuses that number. A citation in chapter 3 to a card
  placed in chapter 7 reads "7.2.8" everywhere.
- **Proofs are transparent.** A proof card is neither numbered nor consumes an
  item slot: theorem 2.3's proof does not use up 2.4.
- **References never count.** `\entryref` is a link; only `\entryblock`
  placement positions a card.

## Numbers are the project's view

Because the coordinate is a function of where a hash first occurs *in that
project's documents*, the same content-addressed card can carry different
numbers in different projects — and stays perfectly consistent within each.
Reorder a section, move a card, or drop in a new one, and everything renumbers
itself at once; the source text of the card, including any numbering it had in
its original book, plays no role.

For how cards are stored and hashed, see the [data model](/docs/data-model);
for the document macros themselves, see [writing with cards](/docs/authoring).
